Mitigations planned: `updateDoc` only for `displayName` and `anniversaryDate`; Firestore `affectedKeys().hasOnly(['displayName', 'anniversaryDate'])`; membership checks against existing `resource.data.memberIds`; client create/delete stays denied; calm error categories separate UI copy from console details.
Block on: any rule allowing client `create` on `users` or `couples`, any client write to `memberIds`, or any settings write path that uses `setDoc` to create/overwrite the couple document.
</threat_model>
